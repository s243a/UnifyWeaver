# Streaming Data Pipelines — Implementation Plan

Phased rollout targeting the enwiki MySQL dump parser as the
reference case. Each phase delivers a working end-to-end pipeline;
later phases swap transport without touching the Prolog.

## Phase S1 — Text IO (existing infrastructure, light additions)

Goal: the enwiki dump parser works end-to-end using only existing
`native_glue.pl` + `pipe_glue.pl` + a small Rust leaf primitive.

### S1.1 — Rust leaf primitive

Hand-written Rust crate at `src/unifyweaver/runtime/rust/mysql_stream/`:

```rust
// src/unifyweaver/runtime/rust/mysql_stream/src/lib.rs
use flate2::read::GzDecoder;
use std::io::{self, BufRead, BufReader, Write};
use std::fs::File;

/// Stream INSERT tuples from a gzipped MySQL dump.
/// Each yielded item is a Vec<Field> where Field is a typed column value.
pub fn iter_category_links(path: &str) -> impl Iterator<Item = (i32, String)> {
    let file = File::open(path).expect("open dump");
    let gz = GzDecoder::new(file);
    let reader = BufReader::with_capacity(1 << 20, gz);
    MysqlInsertIter::new(reader).filter_map(|tuple| {
        // categorylinks columns: (cl_from, cl_to, cl_sortkey, cl_timestamp,
        //                         cl_sortkey_prefix, cl_collation, cl_type)
        match (tuple.get(0), tuple.get(1), tuple.get(6)) {
            (Some(Int(from)), Some(Str(to)), Some(Str(typ))) if typ == "subcat"
                => Some((*from as i32, to.clone())),
            _ => None,
        }
    })
}
```

The `MysqlInsertIter` is the byte-level state machine — ~150
lines of hand-written Rust, not transpiled. It handles:
- Skipping CREATE TABLE preamble until first INSERT
- Walking INSERT VALUES tuples
- String escape decoding (`\'`, `\\`, `\n`, `\r`, `\0`)
- Type dispatch (quoted string vs unquoted integer vs NULL)

### S1.2 — Rust binary wrapper (generated by streaming glue)

The glue generates a pipe-compatible `main()` that imports
`mysql_stream` and writes tuples to stdout. This is the
extension to `generate_rust_tsv_main/2` we noted in the spec:

```rust
// auto-generated src/main.rs
use std::io::{self, Write};
use mysql_stream::iter_category_links;

fn main() {
    let args: Vec<String> = std::env::args().collect();
    let path = &args[1];
    let stdout = io::stdout();
    let mut out = stdout.lock();
    for (child, parent) in iter_category_links(path) {
        writeln!(out, "{}\t{}", child, parent).unwrap();
    }
}
```

### S1.3 — Prolog declarations

The user-facing Prolog is the shape we settled on in the
philosophy doc:

```prolog
:- use_module(library(cross_target_glue/target_mapping)).

:- declare_target(parse_category_links/3, rust,
                  [streaming(true, text),
                   source(file),
                   leaf(true),
                   native_crate(mysql_stream)]).

:- declare_target(ingest_to_lmdb/3, python,
                  [streaming(true, text)]).

process_dump(DumpPath, LmdbPath) :-
    forall(
        parse_category_links(DumpPath, Child, Parent),
        ingest_to_lmdb(LmdbPath, Child, Parent)
    ).
```

The `leaf(true)` option tells the Rust codegen "don't transpile
this — bind it to the `native_crate(mysql_stream)` primitive."

### S1.4 — Python consumer (generated by streaming glue)

The glue generates a Python script that reads stdin and calls
`ingest_to_lmdb` per line:

```python
# auto-generated ingest.py
import sys, lmdb

def ingest_to_lmdb(env, child, parent):
    with env.begin(write=True) as txn:
        txn.put(str(child).encode(), parent.encode())

if __name__ == '__main__':
    lmdb_path = sys.argv[1]
    env = lmdb.open(lmdb_path, map_size=10**10)
    for line in sys.stdin:
        child, parent = line.rstrip('\n').split('\t', 1)
        ingest_to_lmdb(env, int(child), parent)
    env.sync()
```

### S1.5 — Orchestration

`native_glue.pl`'s existing `generate_native_pipeline/3`
composes the Rust binary + Python consumer into a shell command:

```bash
./target/release/parse_category_links $DUMP_PATH | python3 ingest.py $LMDB_PATH
```

No new infrastructure needed for this step — the pipeline
generator already handles the composition.

### S1.6 — Deliverables

- [ ] `src/unifyweaver/runtime/rust/mysql_stream/` — leaf crate
      (tokenizer, handles escape rules, 100% test coverage)
- [ ] `generate_rust_streaming_main/3` in `native_glue.pl` —
      new template that wraps a leaf crate into a pipe-writer
- [ ] `leaf(true)` + `native_crate(CrateName)` option handling
      in the Rust codegen path
- [ ] Example: `examples/streaming/enwiki_category_ingest.pl`

- [ ] Documentation: `examples/streaming/README.md`

### S1.7 — Success criteria

- Parser handles `enwiki-latest-categorylinks.sql.gz` without
  dropping or misparsing tuples (verified against a MariaDB
  import as ground truth on a small fixture)
- End-to-end: `swipl -g "run(process_dump('enwiki.sql.gz',
  './enwiki_lmdb/'))" -t halt` populates LMDB with the full
  subcat hierarchy
- Throughput: at least 10 MB/s of gzipped input on a single
  core (Phase 1 is text-bound; that's fine)

## Phase S1.5 — Binary framing

Goal: demonstrate the transport-swap principle — same Prolog,
different glue template, measurable speedup.

### S1.5.1 — New Rust template

Sibling to `generate_rust_tsv_main/2`:

```prolog
%% generate_rust_binary_pipe_main(+Logic, +FrameSpec, -Code)
%  Emits a Rust main() that writes packed binary frames to stdout.
%  FrameSpec = i32_pair | i64_pair | record(Fields) | ...
generate_rust_binary_pipe_main(Logic, i32_pair, Code) :-
    format(atom(Code), '
use std::io::{self, Write};
use mysql_stream::iter_category_links;

fn main() {
    let args: Vec<String> = std::env::args().collect();
    let stdout = io::stdout();
    let mut out = io::BufWriter::with_capacity(1 << 16, stdout.lock());
    for (child, parent_id) in iter_category_links(&args[1]) {
        out.write_all(&child.to_le_bytes()).unwrap();
        out.write_all(&parent_id.to_le_bytes()).unwrap();
    }
}
', []).
```

Note this version assumes the parent string has been interned
to `i32` already — the `(Int32, String)` shape from S1.1 needs
an intermediate interner, which is itself a separable streaming
stage. One way:

```prolog
process_dump(DumpPath, LmdbPath) :-
    forall(
        (   parse_category_links(DumpPath, Child, ParentStr),
            intern_atom(ParentStr, ParentId)
        ),
        ingest_to_lmdb(LmdbPath, Child, ParentId)
    ).
```

Where `intern_atom/2` is a Python predicate owning the intern
table. In Phase 1.5 this means: Rust writes `(i32, string)`
bytes, Python reads and interns, Python re-writes to LMDB. Two
pipe stages.

Alternative: bring the intern table into Rust via a dedicated
primitive. Moves complexity to the producer side but avoids the
two-stage pipe.

### S1.5.2 — Python consumer update

Same generated shape as S1.4 but reading bytes:

```python
import struct, sys
while True:
    buf = sys.stdin.buffer.read(8)
    if not buf:
        break
    child, parent = struct.unpack('<ii', buf)
    ingest_to_lmdb(env, child, parent)
```

### S1.5.3 — Deliverables

- [ ] `generate_rust_binary_pipe_main/3`
- [ ] Python binary reader template in a streaming-glue module
- [ ] Benchmark comparing S1 vs S1.5 throughput on the enwiki
      dump
- [ ] Frame-spec validation: reject declarations with
      variable-length fields in `i32_pair`/`i64_pair` shapes

## Phase S2 — API / pyo3 (research branch)

Not scoped for initial delivery. Notes captured for the future:

- pyo3 + maturin is the cleanest Rust→Python API path. Producer
  becomes a `#[pyclass]` exposing `__iter__` / `__next__`.
- The leaf primitive from S1.1 is reused unchanged — only the
  wrapping shell changes from "main that writes stdout" to
  "module that exposes a class."
- Build orchestration needs `maturin develop` integrated into
  `native_glue.pl`'s cargo management.
- Error propagation: `anyhow::Error` → `PyErr` via pyo3's
  `Into<PyErr>` impl.
- First milestone: port the S1.5 pipeline to S2, measure
  overhead reduction (expect 10-50x on small records due to
  eliminated IPC + parse).

## Cross-cutting: what does NOT change

Across all three phases, the Prolog stays:

```prolog
:- declare_target(parse_category_links/3, rust,
                  [streaming(true, Mode), leaf(true), ...]).
:- declare_target(ingest_to_lmdb/3, python,
                  [streaming(true, Mode)]).

process_dump(DumpPath, LmdbPath) :-
    forall(parse_category_links(DumpPath, C, P),
           ingest_to_lmdb(LmdbPath, C, P)).
```

Only `Mode` changes: `text` → `binary(i32_pair)` → `api(pyo3)`.
The `forall` composition, the predicate arities, the user-facing
import list — all identical. That's the promise of the
abstraction, and the key success criterion for the design.

## Risks and mitigations

| Risk | Mitigation |
|------|-----------|
| Rust parser authoring takes longer than expected | Start with a subset of escape rules; validate against MariaDB; expand coverage iteratively |
| `leaf(true)` option collides with existing option semantics | Namespace as `streaming_leaf(true)` if conflicts surface in review |
| Pipe buffering stalls on irregular input | Explicit `BufWriter::with_capacity(64KB)` as shown; document the knob |
| Binary framing breaks on big-endian platforms | Use `to_le_bytes` / `from_le_bytes` explicitly, document as LE-only for now |
| Intern table split across languages causes ID mismatches | S1.5 uses two-stage pipe with Python-owned intern; S2 unifies via shared object |

## Ordering with other work

This is a separable effort from the WAM scaling benchmarks. The
enwiki dump is the motivating use case, but S1 can be validated
on the existing simplewiki dump first:

```
simplewiki (90 MB dumps) → parse pipeline → LMDB
  → validated against the existing simplewiki_categories.db output
```

That gives us correctness verification with fast iteration
before committing to the enwiki download.
