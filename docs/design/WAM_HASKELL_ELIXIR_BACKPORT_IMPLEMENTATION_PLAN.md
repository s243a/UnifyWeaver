# WAM Haskell Backport Implementation Plan From Elixir Large-Scale Benchmarks

## Goal

Port the useful WAM-Elixir large-scale benchmark lessons back to WAM-Haskell:

- Avoid calling the WAM/FFI kernel for seeds outside the structural demand set.
- Preserve path-observing behavior while allowing folded aggregate fast paths.
- Use LMDB cache prepopulation deliberately for structural demand sets.

Semantic top-K cache warming is included as a later cross-target planning stage.
It is not specific to Haskell and should be designed so Elixir, Rust, Scala, and
distributed backends can use the same source-level hints.

## Phase 1: Haskell Demand Gate Before Kernel Entry

1. Locate the Haskell seed loop generated by
   `templates/targets/haskell_wam/main.hs.mustache`.
2. Reuse the existing `computeDemandSet` and `filterByDemand` helpers.
3. Before constructing per-seed WAM state or calling `collectForeignSolutions`,
   check whether the seed ID is in `demandSet`.
4. Return `(cat, 0.0)` immediately for seeds outside the demand set.
5. Keep existing demand metrics and add a skipped-seed metric.

Acceptance:

- `50k_cats` and `100k_cats` output hashes stay unchanged.
- Haskell query time drops materially on the category-only workload where
  `demand_set_size=11`.
- The benchmark table clearly names the mode as kernel-enabled FFI.

## Phase 2: Folded Aggregate Kernel Surface

1. Add a Haskell kernel entry that folds hop weights directly for
   aggregate-only `category_ancestor` calls.
2. Keep `collectForeignSolutions` for path-preserving queries.
3. Add planner logic that selects folded mode only when the caller observes an
   aggregate result and does not observe the individual path rows.
4. Add correctness tests comparing:
   - path-preserving enumeration followed by external aggregation
   - folded aggregate kernel result

Acceptance:

- Diamond graph tests preserve duplicate-path contribution.
- Cycle tests preserve visited-list behavior.
- Existing path-observing tests continue to use path-preserving mode.

## Phase 3: Structural Cache Prepopulation

1. Add an LMDB cache prepopulation hook for Haskell edge lookups.
2. For root-bound graph kernels, prepopulate cache entries for nodes in the
   demand set.
3. Keep cache memory bounded using the existing L1/L2 overwrite-on-collision
   policy.
4. Emit `cache_preload_entries`, `cache_hits`, and `cache_misses`.

Acceptance:

- Prepopulation does not change output hashes.
- Memory remains bounded under the configured cache capacity.
- Cache metrics show whether misses remain after structural warming.

## Phase 4: Prolog-Level Planning Declarations For Haskell Backport

1. Add parser/planner support for declarations such as:

   ```prolog
   :- use_kernel(category_ancestor/4, ffi).
   :- use_kernel(category_ancestor/4, demand_filter(root_bound)).
   :- predicate_property(category_ancestor/4,
       graph_kernel(edge_predicate(category_parent/2),
                    direction(child_to_parent),
                    start_arg(1),
                    root_arg(2),
                    visited_arg(4))).
   ```

2. Treat `demand_filter(root_bound)` as semantics-preserving for recognized
   graph kernels.
3. Record the active structural planning declarations in generated benchmark
   output.

Acceptance:

- The generated benchmark records the active planning declarations.
- Missing or unsupported graph-shape declarations produce a clear planning
  fallback instead of silently changing semantics.

## Phase 5: Cross-Target Semantic Top-K Cache Warming

This phase is deliberately cross-target. It should not be implemented only in
Haskell unless the planner interface is target-neutral.

1. Define an embedding index interface that can score candidates against one or
   more endpoints.
2. Implement local top-K cache warming for desktop workloads where the embedding
   index fits locally.
3. Add a distributed plan for larger workloads:
   - shard candidates
   - score candidates locally
   - emit sorted local top-K streams
   - merge streams into global top-K
   - prepopulate LMDB cache for the selected IDs

This is merge-sort/Hadoop-style execution. It should remain optional because
many benchmark and desktop workloads do not require cluster-scale ranking.

Acceptance:

- Cache-only semantic warming does not change output hashes.
- If semantic filtering is used to prune candidates, the run records that it is
  heuristic unless a completeness contract is supplied.
- The benchmark reports top-K size, preload entries, and whether ranking was
  local or distributed.
- The same Prolog declarations can be consumed by at least two targets, or by a
  target plus a standalone preprocessing tool.

## Open Questions

- What is the first practical top-K size for a PC: 50k, 500k, or 5M?
- Which embedding artifact should be the default for Wikipedia categories?
- Should folded aggregate kernels be generated from aggregate recognition in
  Prolog, or declared explicitly per predicate first?
- Should Haskell's existing LMDB cache modes grow a named
  `structural_preload` mode, or should preload be orthogonal to cache mode?
