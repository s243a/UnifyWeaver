# Enwiki MTC Validation-Calibrated Precompute Estimator

This pass closes the loop between the depth precompute planner and the boundary-cache validation rows.  The estimator now accepts boundary-cache JSONL files and can run with `--cap-mode validation`, where each measured boundary depth supplies a conservative saved-per-hit value:

```text
saved_per_hit = max(0, mean(full_time_ns) - mean(cached_time_ns)) / mean(cache_hits)
```

Rows with zero observed cache hits are excluded from saved-per-hit calibration.  They are reported explicitly because, for targets constrained to the ancestor tree with a common root and selected ancestor boundaries, zero-hit rows indicate the benchmark did not exercise the intended boundary condition.

## Result

Validation rows for boundary depths `2..6` have at least one positive-hit row, and those positive-hit rows still produce zero clipped savings.  Depth `7` has no positive-hit rows, so it is marked unusable for validation capping rather than ordinary no-pay evidence.  More generally, any zero-hit row should be read as a validation-selection warning: if the sampled targets are restricted to the ancestor tree and share the root, an ancestor-boundary cache should be hit.

The estimator still recommends depths `0` and `1`, but those rows are not backed by the boundary-cache validation files used here.  Treat them as model-side candidates for a follow-up benchmark, not as measured wins.

One planning caveat became clearer during review: the global parent-branching prior `E[p^2]/E[p]` is probably too pessimistic for common-root ancestor searches.  If materialization and query traversal are restricted to the ancestor tree of the target, the effective branching factor is ancestor-conditioned and likely lower than raw enwiki parent branching.  That means deeper histograms or fitted distributions may be worth materializing than the global prior alone would suggest.  Future estimator runs should prefer a depth-conditioned ancestor-tree branching prior over the raw global parent-branching statistic.  A concrete preprocessing target is to recompute `E[p^2]/E[p]` after excluding every category node that does not have `Category:Main_topic_classifications` as an ancestor, and then counting only retained parents for each retained node.  That root-conditioned parent branching should be generated by the ingest/profile pipeline and passed into the estimator as a prior; the online query path should not discover the root-conditioned subgraph itself.  The preprocessing work is still part of the same amortization decision: its cost only pays off when expected query volume over the dataset is high enough.

## Implication

For capped smoke searches, precomputing ancestor histograms does not pay once the search still hits the same expansion cap and pays decode/splice overhead.  The next useful validation should either benchmark shallower root-adjacent caches directly, or run a less-capped workload where cache hits can actually reduce suffix traversal.  It should also constrain sampled targets to the ancestor tree so zero-hit rows become an invariant failure instead of an ambiguous timing datum, then recompute the planning prior from that constrained ancestor subgraph.  The next preprocessing/profile pass should report both raw parent branching and root-conditioned parent branching, plus the preprocessing cost, so the estimator can show how much the common-root filter lowers expected histogram growth and how many dataset queries are needed to amortize that work.
