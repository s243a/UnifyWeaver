# WAM-Elixir Tier-2 findall: parallel vs sequential

## TL;DR

For a Tier-2-eligible 8-clause predicate, the parallel/sequential
crossover depends heavily on per-branch workload shape:

- **Atom-only nested findall (K-solution inner enumeration per branch)**:
  parallel beats sequential at **K ≈ 100–200**. At K=1000, parallel is
  ~3.65×.
- **Arithmetic recursion (`iterate(N) :- N > 0, ..., iterate(N - 1)`)**:
  parallel beats sequential at **N ≈ 10–50** — roughly an order of
  magnitude lower than the nested-findall crossover. At N=1000,
  parallel is ~3.51×.

Heavier per-iteration cost (arithmetic = comparison + arith eval +
trail entry per step) amortises `Task.async_stream` spawn overhead at
smaller workload sizes than the fact-only inner-findall pattern.

## Shape 1 — nested_findall (atom-only inner enumeration)

8-clause `bench_p`. Each clause body runs a nested
`findall(_, inner_p(_), _)` over a K-solution `inner_p`. Per-branch
work scales linearly with K.

| K (inner solutions / branch) | Parallel (μs, median) | Sequential (μs, median) | Speedup (seq / par) |
| ---: | ---: | ---: | ---: |
|    1 |   190 |    63 | 0.33× |
|   10 |   421 |   289 | 0.69× |
|   50 |  1244 |  1092 | 0.88× |
|  200 |  2353 |  4329 | **1.84×** |
| 1000 |  6867 | 25075 | **3.65×** |

## Shape 2 — arith (recursive integer countdown)

8-clause `bench_p`, each clause body calls `iterate(N)` where
`iterate(0).` and `iterate(M) :- M > 0, M1 is M - 1, iterate(M1)`.
Per-branch work scales linearly with N. This shape was infeasible in
the original benchmark because of two upstream bugs (integer head-
match and the missing `>/2` builtin); now that #1776 and #1777 are
merged, arithmetic-heavy workloads run correctly.

| N (countdown depth / branch) | Parallel (μs, median) | Sequential (μs, median) | Speedup (seq / par) |
| ---: | ---: | ---: | ---: |
|    1 |    264 |     119 | 0.45× |
|   10 |    851 |     756 | 0.89× |
|   50 |   4402 |    9547 | **2.17×** |
|  200 |  32229 |   84870 | **2.63×** |
| 1000 | 595085 | 2089435 | **3.51×** |

Hardware: 4 vCPU Intel Xeon @ 2.10 GHz, BEAM `System.schedulers_online() = 4`.
Numbers are 10-run medians per cell, post-warmup, on a single elixir
invocation per cell.

## Methodology

Generated by `examples/benchmark/wam_elixir_tier2_findall_benchmark.pl`:

- `bench_shapes/1` selects which shape variants to run.
- For each `(shape, mode, work)` cell: compile a fresh project, run
  `findall(X, bench_p(X), L)` 10 times back-to-back after one warmup,
  capture per-run elapsed_us via `:timer.tc/1`, report the median.
- Sequential mode: `intra_query_parallel(false)` in project options.
  Parallel mode: option omitted (default ⇒ super-wrapper fans out via
  `Task.async_stream`).

## Reading the numbers

- Both shapes show **the same qualitative pattern**: parallel loses at
  small workloads (process-spawn + send/receive overhead dominates),
  parallel wins past a workload-shape-specific crossover, and at the
  maximum measured workload both shapes converge to ~3.5× — close to
  the 4× theoretical max for this hardware.
- **Arith crossover is ~10× smaller** than nested-findall crossover.
  Each `iterate` step does more work than each fact-only inner-findall
  iteration: `>/2` builtin + `is/2` (heap put_structure + arith eval)
  + tail-call into iterate (try_me_else CP push, get_constant fail,
  backtrack to clause 2, allocate/deallocate). That per-step cost
  amortises spawn overhead at smaller N.
- **Absolute cost differs by ~85×.** Per "size" unit, arithmetic
  recursion is far more expensive than fact-only enumeration on this
  target — partly real per-step compute, partly inefficiency in the
  WAM-interpreted Elixir runtime (CP-stack growth from try_me_else on
  every recursive call, eval_arith heap walk).

## Implications

- **`forkMinBranches=3` (clause-count purity gate) is necessary but
  not sufficient.** A correctness gate doesn't consider per-branch
  cost, so Tier-2 will be slower than sequential for low-cost
  workloads — including the fact-only predicates in Phase 4c's
  correctness suite (which exists to test correctness, not speed).
  Future work: a cost-aware gate that estimates per-branch cost
  statically, or a runtime probe on the first invocation.
- **Crossover is workload-shape-specific.** The two shapes here
  differ by ~10× in crossover point. Real-world workloads will fall
  somewhere in between; a single tuning constant won't be optimal.
- **Both shapes top out at ~3.5×** on this 4-vCPU box (theoretical
  max 4×). Remaining gap = `Task.async_stream` baseline cost +
  parent-side `merge_into_aggregate` flat-map. Likely improves on
  more cores; would also improve with batched merging if BEAM had
  cheap shared-state aggregation primitives.

## Reproducing

```sh
swipl -g run_benchmark -t halt examples/benchmark/wam_elixir_tier2_findall_benchmark.pl
```

To run only one shape: edit `bench_shapes/1` in the benchmark file.
Skipped gracefully when `elixir` is not on `PATH`.

## History / changelog

- **Initial measurement** (PR #1774): atom-only `nested_findall`
  shape only. The arith shape was infeasible at the time because of
  the integer head-match bug — `iterate(0)` returned `:fail` and the
  countdown never executed.
- **This update**: arith shape added. Crossover comparison shows
  per-step cost dominates the crossover threshold.
