# WAM-Elixir Tier-2 findall: parallel vs sequential

## TL;DR

For a Tier-2-eligible 8-clause predicate where each clause body does a
nested K-solution sequential findall, **parallel mode beats sequential
once per-branch work exceeds ~K=100**. At K=1000 parallel is ~3.7×
faster.

| K (inner solutions / branch) | Parallel (μs, median) | Sequential (μs, median) | Speedup (seq / par) |
| ---: | ---: | ---: | ---: |
|    1 |   212 |    91 | 0.43× |
|   10 |   458 |   244 | 0.53× |
|   50 |  1350 |  1099 | 0.81× |
|  200 |  2330 |  5104 | **2.19×** |
| 1000 |  6521 | 24372 | **3.74×** |

Hardware: 4 vCPU Intel Xeon @ 2.10 GHz, BEAM `System.schedulers_online() = 4`.
Numbers are 10-run medians per cell, post-warmup, on a single elixir
invocation per cell.

## Methodology

Generated by `examples/benchmark/wam_elixir_tier2_findall_benchmark.pl`:

- 8-clause Tier-2-eligible `bench_p/1`. Each clause's body runs a
  nested `findall(_, inner_p(_), _)` over a K-solution `inner_p/1`.
- Driver: `findall(X, bench_p(X), L)` invoked K=10 times back-to-back
  per (mode, work) cell after one warmup call.
- Two project variants compiled from the same Prolog source: one with
  `intra_query_parallel(false)` (sequential), one without (default ⇒
  parallel; super-wrapper fans out via `Task.async_stream`).
- Per-iteration timing via `:timer.tc/1`.

## Reading the numbers

- **K ≤ 50**: parallel is slower. `Task.async_stream` overhead
  (process spawn + send/receive + Enum.flat_map merge) dominates the
  per-branch compute.
- **K ~ 100**: crossover region. Parallel and sequential are within
  noise of each other.
- **K ≥ 200**: parallel wins. At K=200 the speedup is ~2.2×; at K=1000
  it approaches ~3.7× — close to the 4× theoretical max for this
  4-scheduler box. The remaining gap is `Task.async_stream`'s baseline
  cost plus the parent's `merge_into_aggregate` flat-map.

## Implications

- **Tier-2's `forkMinBranches=3` clause-count threshold is necessary
  but not sufficient.** A 3+ clause Tier-2-eligible predicate with
  near-trivial bodies will be slower in parallel mode than sequential.
  Phase 4's purity gate is conservative on correctness but doesn't
  consider per-branch cost.
- **A future cost-aware gate** could improve average performance by
  staying sequential when expected per-branch work is below the
  crossover threshold. Approximating per-branch cost statically is
  hard, but a runtime probe (count CPs / heap allocations on the
  first invocation, switch modes for subsequent invocations) is
  feasible. Out of scope for Phase 4.
- **The crossover is workload-shape-specific.** Per-branch work that
  allocates heap (deep terms, lots of unification) will likely have a
  lower crossover because each iteration costs more than the simple
  fact-only `inner_p` here. Conversely, fact-only Tier-2 predicates
  (Phase 4c's `phase4c_p`/`phase4c_pn`) sit firmly in the
  parallel-loses regime — those tests assert correctness, not speed,
  which is consistent with the data above.

## Reproducing

```sh
swipl -g run_benchmark -t halt examples/benchmark/wam_elixir_tier2_findall_benchmark.pl
```

Skipped gracefully when `elixir` is not on `PATH`.

## Side finding (out of scope for this benchmark)

The first iteration of this benchmark used a recursive countdown
`iterate(N) :- N > 0, N1 is N - 1, iterate(N1)` per branch. Every
workload size completed in roughly the same wall-clock time —
suspicious. Direct probing showed `iterate/1` returning `:fail`
even for `iterate(0)`. Inspection of the lowered Elixir revealed
the WAM-Elixir compiler emits `val == "0"` (string) in
`get_constant 0, A1`'s lowering, instead of `val == 0` (integer):
integer constants in clause heads are stringified during codegen,
which breaks numeric head-match for any predicate that pattern-
matches an integer literal.

This is a pre-existing compiler bug in
`src/unifyweaver/targets/wam_elixir_target.pl`'s lowering of
`get_constant` (likely in `quote_wam_constant/2` or its caller).
Out of scope for this benchmark — recorded here for follow-up
because the integer-as-string emission means any benchmark using
arithmetic-heavy workloads will silently no-op. The atom-only
workload above sidesteps it entirely.
